Architecture Summary

Goal: Implement a reliable transport mechanism over a flaky UDP network that minimizes playout latency while adhering to a strict bandwidth cap (≤ 2.0x overhead) and a deadline miss rate cap (≤ 1.0%).

1. Design Choices

To achieve the lowest possible delay, Forward Error Correction (FEC) was chosen over ARQ (Retransmissions).

In a hostile network where round-trip times (RTT) can exceed 160ms (due to 80ms one-way jitter), relying on NACKs and retransmissions forces the playout buffer to be artificially large, significantly penalizing the latency score. FEC eliminates the need for round-trip feedback by proactively sending redundant data.

2. Piggybacked FEC Architecture

The chosen FEC strategy is Piggybacking. Each frame's packet transmitted by the sender contains both the current frame payload and the previous frame payload.

  • Sender: At time t0 + i*20ms, the sender receives frame i. It immediately transmits a UDP packet containing: seq, payload[i], and payload[i-1].
  • Receiver: Upon receiving a packet, it extracts both frames. If either frame hasn't been seen yet, it is immediately forwarded to the harness player.

If frame i-1 is dropped by the relay, it is recovered instantly when frame i arrives, adding only 20ms of intrinsic recovery delay instead of a full 160ms RTT.

3. Beating the Bandwidth Limit

A naive implementation of Piggybacked FEC fails the assignment constraints. A raw frame is 164 bytes (4 bytes seq + 160 bytes payload). A piggybacked packet is 324 bytes (4 bytes seq + 160 bytes current + 160 bytes previous). This yields an overhead of 324 / 160 = 2.025x, which violates the ≤ 2.0x cap.

The Innovation: The sender selectively drops the redundant payload on every 12th packet. 11 packets are 324 bytes, and 1 packet is 164 bytes. This brings the average bandwidth overhead down to exactly 1.94x. The final Profile B validation at 100ms had a 0.67% miss rate, which comfortably passes the 1.0% cap limit.

4. System-Level Receiver Behavior

At the system level, the receiver operates in a purely event-driven manner. It does not manage an internal threaded jitter-buffer queue. The harness player judges frames based on when they hit the 47020 port relative to the absolute deadline.

By extracting both the primary and redundant frames upon `recvfrom` and instantly dispatching them to the player via `sendto`, the receiver avoids all processing delays and thread context-switching overhead, ensuring packets meet the tightest possible deadline.
